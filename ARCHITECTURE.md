# Архитектура ЕППБ

> Этот документ — единый источник правды по структуре проекта.

## Обзор системы
ЕППБ состоит из трёх зон:
- Публичная часть (главная, каталог услуг, карточка услуги)
- Личный кабинет пользователя (заявки, статусы, документы, уведомления)
- Административный кабинет (конструктор услуг, управление заявками)

Ключевой элемент — no-code конструктор, через который создаются услуги.
Рендерер превращает JSON-конфиг услуги в рабочую пошаговую форму.

## Модель данных

Модель описана как TypeScript-типы в папке `/types` (по файлу на смысловой блок,
всё реэкспортируется из `types/index.ts` — импорт: `import { Service } from "@/types"`).
Валидация и исполнение логики — в `/lib` (см. ниже). Этот раздел — словесное
описание; при изменениях правим **и типы, и этот раздел**.

### Сущности

| Сущность | Файл | Что описывает |
|----------|------|---------------|
| `Service` | `types/service.ts` | Услуга — JSON-конфиг, который рендерер превращает в пошаговую форму. |
| `Step` | `types/service.ts` | Шаг (этап) услуги — один экран мастера с полями. |
| `Field` | `types/field.ts` | Поле формы — самый мелкий кирпичик конструктора. |
| `VisibilityRule` | `types/logic.ts` | Логика ветвления — показать/скрыть поле или шаг по условию. |
| `CalculatedFormula` | `types/logic.ts` | Расчётное поле — значение вычисляется из других полей. |
| `Reference` | `types/reference.ts` | Справочник — переиспользуемый список значений (опций). |
| `Application` | `types/application.ts` | Заявка — заполненный пользователем экземпляр услуги. |
| `User` | `types/user.ts` | Пользователь и его роль (`user` / `admin`). |

### Два идентификатора поля: `id` и `key`
У каждого `Field` есть **оба**:
- **`id`** — стабильный UUID. Идентичность поля в БД, для связей и истории.
- **`key`** — человекочитаемый машинный ключ (напр. `loanAmount`). **По нему**
  адресуются ответы в `Application.formData`, ссылаются условия и формулы.

Почему так: `key` делает конфиг и формулы читаемыми (`loanAmount * rate / 100`
вместо `f3 * f4 / 100`) и остаётся стабильным при смене `label`, а `id` даёт
жёсткую идентичность на уровне БД. `key` уникален в пределах услуги.

### Как это связано
- `Service` → массив `Step` → массив `Field`.
- `Field` берёт варианты выбора либо из `options`, либо из справочника по
  `referenceId` (Reference.id). Может иметь `visibilityCondition` (показ по условию)
  и/или, при `type: "calculated"`, — `formula`.
- Ответы пользователя лежат в `Application.formData` под ключами `Field.key`.
- Условия и формулы адресуют поля тоже по `Field.key`.
- `Application.serviceVersion` фиксирует версию услуги на момент подачи, чтобы
  поданная заявка не «поехала» при последующем редактировании услуги.

### Ключевой принцип: логика — это данные, а не код
Ветвление (`VisibilityRule`) и расчёты (`CalculatedFormula`) описываются
**структурами данных**, которые исполняет общий движок. Поэтому **одна и та же
JSON-схема** описывает и простую услугу, и сложную многоэтапную с ветвлением и
расчётами — без единой строчки нового кода. Ни одна услуга не хардкодится
отдельной формой.

**Операторы условий:** `equals`, `notEquals`, `greaterThan`, `greaterThanOrEqual`,
`lessThan`, `lessThanOrEqual`, `in`, `contains`, `isEmpty`, `isNotEmpty`.
Условия можно объединять в группы `and` / `or` (в т.ч. вложенные):
```json
{ "logic": "and", "conditions": [
  { "field": "region", "operator": "equals", "value": "almaty" },
  { "field": "loanAmount", "operator": "greaterThan", "value": 50000000 }
] }
```

### Слой исполнения (`/lib`)
- `lib/logic.ts` — исполнитель условий (`isVisible`, `evaluateRule`).
- `lib/formula.ts` — безопасный вычислитель формул без `eval`
  (`evaluateFormula`, `applyCalculations`).
- `lib/schemas/*` — Zod-схемы модели; `serviceSchema` проверяет конфиг услуги
  (уникальность `key`, ссылки условий/формул на существующие поля, синтаксис формул).
- `lib/form-schema.ts` — `buildFormSchema(service)` строит Zod-схему под
  `formData` конкретной услуги (условная обязательность: `required` действует
  только когда поле видно).

### Жизненный цикл заявки (двухфазная подача)
`draft → awaiting_documents → submitted → in_review → approved | rejected`.

- **draft** — черновик заполняется; автосохраняется на каждом шаге (можно уйти и
  вернуться, продолжив с места остановки — `Application.currentStepId`).
- **awaiting_documents** — подана **первичная** заявка (получен номер), но нужны
  документы и согласия. Пользователь видит её в кабинете как «Требуются документы»
  и может догрузить документы позже.
- **submitted** — документы предоставлены, заявка подана полностью (далее `in_review`
  и т.д. — уже действия оператора).

**Как услуга делится на два этапа (без хардкода):** этап документов — это хвостовые шаги
услуги, у которых все поля имеют тип `file` или `checkbox` (документы и согласия);
остальные шаги — первичный этап (`lib/application-stages.ts`, `splitStages`). Если такого
хвоста нет, услуга подаётся в одну фазу (сразу `submitted`).

**Гарантии на сервере (`lib/application-actions.ts`):** при подаче любого этапа сервер не
доверяет клиенту — берёт услугу из БД по `service_id`, пересчитывает `calculated`-поля
(`applyCalculations`) и валидирует данные `buildFormSchema` до смены статуса; на этапе
документов первичные данные неизменны (мёржатся только ключи полей этапа документов).

**RLS:** владелец может менять свою заявку только в статусах `draft` и `awaiting_documents`;
после `submitted` правки закрыты. Файлы полей `file` лежат в приватном бакете
`application-documents` по пути `<userId>/<applicationId>/<fieldKey>/<файл>` — доступ по
первому сегменту пути (id владельца).

### Пример 1 — простая услуга
Одноэтапная услуга без ветвления и расчётов: ФИО и регион из справочника.
```json
{
  "id": "svc-ip-reg",
  "slug": "spravka-ip",
  "title": "Справка о регистрации ИП",
  "organization": "Госкорпорация «Правительство для граждан»",
  "status": "published",
  "version": 1,
  "steps": [
    {
      "id": "step-1",
      "title": "Заявитель",
      "order": 1,
      "fields": [
        { "id": "f-1", "key": "fullName", "type": "text", "label": "ФИО", "required": true },
        { "id": "f-2", "key": "region", "type": "select", "label": "Регион",
          "required": true, "referenceId": "ref-regions" }
      ]
    }
  ],
  "createdAt": "2026-07-07T09:00:00Z",
  "updatedAt": "2026-07-07T09:00:00Z"
}
```
Заявка по ней (`formData` — по `key`):
```json
{
  "id": "app-1",
  "serviceId": "svc-ip-reg",
  "serviceVersion": 1,
  "userId": "user-7",
  "currentStepId": "step-1",
  "status": "submitted",
  "formData": { "fullName": "Иван Иванов", "region": "almaty" },
  "documents": [],
  "statusHistory": [
    { "status": "submitted", "changedBy": "user-7", "changedAt": "2026-07-07T10:00:00Z" }
  ],
  "createdAt": "2026-07-07T09:40:00Z",
  "updatedAt": "2026-07-07T10:00:00Z"
}
```

## Слой интеграций

`lib/integrations/` — **единая точка подключения к Единой интеграционной шине** холдинга.
Сейчас за ней стоят mock-реализации; позже они заменяются реальными вызовами шины
**без правок в UI**, потому что весь портал ходит к интеграциям только через этот слой
(и через тонкие server actions), а не напрямую к внешним системам.

Контекст (ТЗ п.4): обмен планируется через шину; авторизация — eGov IDP; подпись — ЭЦП
(НУЦ РК); заявки уходят в BPM-системы дочерних организаций; уведомления — внешние каналы.

Принципы слоя:
- **Server-only**: модули помечены `import "server-only"` — «шина» доступна только серверу
  (реальные вызовы требуют секретов и не должны попадать в браузер). Чистые ТИПЫ вынесены
  в `lib/integrations/types.ts` (без server-only), чтобы их мог использовать и клиент.
- **Единый интерфейс**: каждый сервис — функция с типизированным входом/выходом. Сигнатуры
  не изменятся при замене mock→реальность.
- Каждый mock помечен `// MOCK`, имитирует сетевую задержку и (в UI) сопровождается бейджем
  «демо-интеграция», чтобы было видно, что это имитация, а не реальное подключение.

Сервисы:
- **Проверка ИИН/БИН** (`bin-registry.ts`) — по номеру возвращает реквизиты организации из
  демо-реестра. UI вызывает через server action `checkCompanyByBin`; на этапе 1 подачи
  кнопка «Проверить» предзаполняет наименование/регион/вид деятельности.
- **Передача в BPM** (`bpm.ts`) — при финальной подаче имитирует отправку во внешнюю
  BPM-систему и возвращает внешний номер (`externalRef`).
- **Проверка/подпись ЭЦП** (`ecp.ts`) — имитирует подписание заявки, возвращает `signedAt`.
- **Уведомления** (`notifications.ts`) — единая точка внешних уведомлений (SMS/email через
  шину); внутренние уведомления кабинета (из `status_history`) — отдельный механизм.

Фасад `submitApplicationToBus` (`index.ts`) объединяет подпись → BPM → уведомление и
вызывается из серверного экшена подачи (`lib/application-actions.ts`). Результаты
записываются в заявку (`externalRef`, `signedAt`) и в историю статусов.

## AI-слой

`lib/ai/` — **единственная точка вызова Anthropic API** из всего портала. Как и слой
интеграций, он **server-only** (`import "server-only"`): ключ `ANTHROPIC_API_KEY` и SDK
никогда не попадают в браузер. Модель — **Claude Haiku** (`claude-haiku-4-5-20251001`,
самая дешёвая) для экономии кредитов. UI ходит к AI только через тонкие server actions
(`lib/ai-actions.ts`), а не напрямую. Исключение по модели — генерация структуры услуги:
там нужен надёжный валидный JSON нашей схемы, поэтому используется **Claude Sonnet**
(`claude-sonnet-5`, константа `AI_MODEL_STRUCTURE`); операция редкая (автор-админ), цена
незначима. `generateText` принимает необязательный `model`.

**Мягкая деградация.** `generateText` (`lib/ai/client.ts`) никогда не бросает исключение:
любой сбой (нет ключа, сеть, лимиты API, пустой ответ) превращается в
`{ ok: false, reason: "unavailable" | "empty" }` (тип `AiResult<T>`). AI — прогрессивное
улучшение: страницы рендерятся и работают без него, при сбое показывается аккуратное
сообщение. Каждый AI-ответ в UI помечен бейджем «Сгенерировано AI» (`components/ai/ai-badge`).

Реализованные сценарии (оба — на РЕАЛЬНЫХ данных услуг из БД, не на хардкоде):

- **Подбор услуги по описанию ситуации** (`lib/ai/recommend.ts`; UI —
  `components/ai/service-finder.tsx` на главной и в каталоге). Пользователь описывает
  ситуацию словами → модель выбирает 1–3 услуги из переданного каталога
  (`getPublishedServices`). Две гарантии честности: (1) промпт запрещает выдумывать услуги и
  требует пустой ответ, если ничего не подходит; (2) **независимо от модели** результат
  фильтруется в коде — остаются только `slug`, реально присутствующие в каталоге.
- **Объяснение услуги простым языком** (`lib/ai/explain.ts`; UI —
  `components/ai/explain-service.tsx` на карточке услуги). Кнопка «Объяснить простыми
  словами» → модель переформулирует условия конкретной услуги (описание +
  `getServiceFacts`) для предпринимателя без юробразования.
- **Проверка заявки перед отправкой** (`lib/ai/review.ts`; UI —
  `components/ai/application-review.tsx`, встраивается в путь подачи через необязательный
  проп `reviewSlot` рендерера на последнем шаге). Контекст модели строится детерминированно
  из конфига: только ВИДИМЫЕ (с учётом ответов, `isVisible`) не-расчётные поля с их типами,
  ограничениями и значениями. Модель (Haiku) возвращает список замечаний простым языком.
  Это **помощь, а не блокировка**: формальную валидацию по-прежнему делает `buildFormSchema`
  (Zod) при отправке — AI её дополняет. Услуга и фаза берутся из БД на сервере, клиенту не
  доверяем.
- **Генерация структуры услуги по описанию** (`lib/ai/generate-service.ts`; UI —
  `components/ai/generate-service.tsx` в конструкторе). Автор пишет услугу словами → модель
  (Sonnet) выдаёт JSON нашей схемы (этапы+поля, ветвление, формулы). Сервер проставляет `id`,
  затем **валидирует через `serviceConfigSchema`** — ту же схему, что и публикация. В
  конструктор попадает только валидный черновик; некорректный ответ не роняет страницу
  (`reason: "invalid"` + понятное сообщение, предложение повторить). Черновик открывается для
  ручной доработки и **не публикуется автоматически**. Экшен доступен только админу (роль
  проверяется явно, т.к. генерация не пишет в БД и RLS её не прикрывает).

**Почему это масштабируемость (критерий 9).** Генерация структуры из текста радикально
ускоряет заведение новых услуг: вместо ручной сборки десятков полей автор описывает услугу
словами и получает готовый черновик за один шаг — это прямой путь к каталогу на 70+ услуг и
новым организациям. При этом `serviceConfigSchema` гарантирует, что no-code модель данных не
ломается: в конструктор попадает только корректная структура, а публикует её человек после
проверки. AI ускоряет создание, схема и человек охраняют качество.

## Витрины данных (карта проектов и аналитика)

Две публичные витрины поверх «задельных» таблиц из init-схемы (RLS: public read / admin write).
Обе — только показ данных, а не отдельные системы.

- **Карта проектов** (`app/projects`, `components/projects/*`, `lib/projects.ts`, таблица
  `projects`). Проекты холдинга на карте Казахстана (Leaflet + OpenStreetMap, без API-ключей).
  Карта грузится **только на клиенте** (`dynamic(..., { ssr: false })`) — Leaflet трогает
  `window` на импорте. Фильтры (организация/регион/отрасль/статус/год) и распределение по
  регионам считаются в памяти из одного отфильтрованного списка. Маркеры — свой SVG-пин,
  цвет по статусу.
- **Аналитика и отчётность** (`app/analytics`, `components/analytics/*`, `lib/analytics.ts`,
  таблица `analytics_items`). **Витрина-каталог** готовых материалов дочерних организаций
  (годовые/финансовые отчёты, дашборды, исследования, обзоры) — по ТЗ 6.7 свои BI/дашборды не
  строим. Доступ к материалу: `access_type = 'link'` (внешняя ссылка, новая вкладка) или
  `'embed'` (встраивание через `iframe` на странице `/analytics/[id]`). Кросс-доменную
  блокировку встраивания (X-Frame-Options) из JS достоверно не поймать, поэтому фолбэк-ссылку
  «открыть в новой вкладке» показываем всегда. Фильтры — по виду материала и организации.

## Масштабируемость
_(заполняется на день 5, промпт 5.2)_
Принципы масштабирования на 70+ услуг, новые организации, роли, интеграции.
